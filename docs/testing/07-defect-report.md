# 缺陷文档

## 1. 缺陷报告模板

### 1.1 标准模板

```markdown
---
id: BUG-XXX
title: [简要描述缺陷]
module: [graph | knowledge_base | conversations | frontend | api | loaders | metrics]
severity: [S0-S4]
priority: [P0-P3]
status: [open | in_progress | resolved | closed | wontfix]
environment: [dev | test | staging | prod]
found_at: [YYYY-MM-DD]
found_by: [测试类型]
---

## 前置条件

[描述测试前的环境/数据准备工作]

## 复现步骤

1. [步骤 1]
2. [步骤 2]
3. [步骤 3]

## 期望结果

[描述应该发生的行为]

## 实际结果

[描述实际发生的行为]

## 附件

- 日志: [路径或内容]
- 截图: [路径]
- 测试数据: [相关数据]

## 根因分析

[可选: 定位到的根因]

## 修复方案

[可选: 建议的修复方式]
```

### 1.2 严重程度定义

| 级别 | 标签 | 定义 |
|------|------|------|
| S0 | 致命 | 系统崩溃、数据丢失、核心功能完全不可用 |
| S1 | 严重 | 主要功能不可用，无变通方案 |
| S2 | 中等 | 功能可用但不符合预期，有变通方案 |
| S3 | 轻微 | 非功能性缺陷（UI 错位、拼写错误、性能） |
| S4 | 建议 | 改进建议，非缺陷 |

### 1.3 优先级定义

| 级别 | 定义 |
|------|------|
| P0 | 立即修复（阻塞发版） |
| P1 | 高优先级（本迭代修复） |
| P2 | 中优先级（下迭代修复） |
| P3 | 低优先级（待排期） |

## 2. 缺陷处理流程

```
发现缺陷 → 提交 Bug 报告 → 开发评审 → 分配责任人 
  → 修复代码 → 代码审查 → 验证测试 → 关闭
```

### 各阶段说明

| 阶段 | 负责方 | 输出 |
|------|--------|------|
| 发现 | 测试人员 | Bug 报告 |
| 评审 | PM + 开发主管 | 确认/拒绝/延后 |
| 分配 | 开发主管 | Assignee |
| 修复 | 开发 | Commit + PR |
| 审查 | 开发 | Code Review |
| 验证 | 测试人员 | 测试通过/不通过 |
| 关闭 | 测试人员 | 关闭 Bug |

## 4. 缺陷分类

| 分类 | 说明 |
|------|------|
| 功能缺陷 | 功能不满足需求规格 |
| 性能缺陷 | 响应慢、内存泄漏、资源未释放 |
| 安全缺陷 | XSS、SQL 注入、敏感信息泄露 |
| UI 缺陷 | 样式错乱、布局异常、文字错误 |
| 兼容性缺陷 | 浏览器/操作系统兼容问题 |
| 文档缺陷 | 文档错误、缺失、过时 |
| 测试缺陷 | 测试用例错误、mock 不准确 |
