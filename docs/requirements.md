# KnowBase 需求基线

## 1. 范围

KnowBase 是一个本地优先的知识库问答助手，采用 React 前端和 FastAPI 后端。当前版本的重点不是单纯演示检索链路，而是提供一套可持续迭代的 RAG 产品骨架：文档导入、检索增强、多工作区、来源追踪、调试面板和运行指标。

本文档描述仓库当前架构下的产品需求和工程要求，用来替代早期基于 Streamlit 方案的旧需求说明。

## 2. 产品目标

- 允许用户把本地文件或公开 URL 导入到知识库中。
- 让用户获得带来源依据的回答，而不是黑盒式生成结果。
- 保留对话历史和书签，支持跨会话持续工作。
- 向开发者暴露足够的调试与指标信息，用来分析检索质量和失败原因。
- 保持本地安装和启动成本尽量低，便于演示和迭代开发。

## 3. 目标用户

### 终端用户

- 针对已导入或预置文档进行问答。
- 查看引用来源和证据质量。
- 按工作区、对话和书签管理知识工作流。

### 开发者与评估者

- 在本地运行完整前后端。
- 检查召回、重排、质量检查和日志输出。
- 在不依赖隐式约定的前提下扩展导入、路由或界面行为。

## 4. 功能需求

### 4.1 对话与回答

| ID | 需求 | 优先级 |
|----|------|--------|
| FR-1 | 系统必须通过 SSE 提供流式回答。 | P0 |
| FR-2 | 在存在证据时，回答必须附带来源元数据。 | P0 |
| FR-3 | 用户必须能够继续多轮对话。 | P0 |
| FR-4 | 系统必须支持多种检索策略：`fast`、`balanced`、`high_quality`、`deep`。 | P1 |
| FR-5 | 质量检查失败时，系统应支持重试或兜底策略。 | P1 |

### 4.2 知识导入

| ID | 需求 | 优先级 |
|----|------|--------|
| FR-6 | 用户必须能够上传受支持的本地文档。 | P0 |
| FR-7 | 用户必须能够导入公开 URL 内容。 | P1 |
| FR-8 | 当后端启动且已配置必要 API Key 时，预置文档应自动加载。 | P1 |
| FR-9 | 用户必须能够清空知识库或删除单个来源。 | P1 |

当前支持的文档格式：

- `.txt`
- `.md`
- `.pdf`
- `.docx`
- `.html`

### 4.3 工作区与对话管理

| ID | 需求 | 优先级 |
|----|------|--------|
| FR-10 | 系统必须支持工作区的增删改查。 | P0 |
| FR-11 | 对话必须可持久化，并能按工作区检索。 | P0 |
| FR-12 | 用户必须能够重命名、导出和删除对话。 | P1 |
| FR-13 | 用户必须能够保存书签。 | P1 |

### 4.4 可观测性与调试

| ID | 需求 | 优先级 |
|----|------|--------|
| FR-14 | 开发者必须能够在 UI 中查看 RAG 调试信息。 | P1 |
| FR-15 | 后端必须通过 API 暴露查询日志和运行指标。 | P1 |
| FR-16 | 系统应提供热点片段和知识库统计信息。 | P2 |

## 5. RAG 工作流要求

后端工作流由 LangGraph 编排，当前架构应保留以下阶段：

1. 问题路由：LLM 分类加规则兜底。
2. 查询改写：必要时结合会话上下文重写用户问题。
3. 候选检索：向量召回与 BM25 词法召回配合使用。
4. 候选融合：对召回结果进行融合或优先级排序。
5. 条件式重排：在高质量策略下执行额外精排。
6. 生成回答：结合来源和邻近 chunk 上下文生成答案。
7. 质量检查：失败时触发重试、扩检索或可选联网搜索兜底。

## 6. API 与 UI 要求

### API

- 后端必须提供聊天、文档、对话、知识库浏览、工作区、书签、指标和健康检查相关 REST 接口。
- 聊天接口必须输出与前端 SSE 解析器兼容的事件流。
- 当 `API_KEY` 已配置时，受保护路由必须启用 Bearer Token 鉴权。

### UI

- 前端必须提供聊天、知识库浏览和指标面板视图。
- 用户必须能够在应用内完成文档导入和来源查看。
- 界面应同时支持桌面端与移动端布局。
- 主题切换能力应保留。

## 7. 数据与存储要求

- 向量数据必须通过本地 Chroma 持久化。
- 对话和工作流状态必须支持重启后恢复。
- 运行期日志和生成产物不得进入版本控制。
- 相对路径配置应以 `backend/` 为解析基准。

当前运行期持久化数据包括：

- `backend/data/chroma_db/`
- `backend/data/checkpoints.db`
- `data/conversations.db`
- `data/rag_logs/`

## 8. 配置要求

后端配置必须从 `backend/.env` 加载。

完整回答能力的必需项：

- `SILICONFLOW_API_KEY`

常用可选项：

- `API_KEY`
- `TAVILY_API_KEY`
- `LANGSMITH_TRACING`
- `LANGSMITH_API_KEY`
- `LLM_MODEL`
- `EMBEDDING_MODEL`
- 检索调优参数，例如 `CHUNK_SIZE`、`TOP_K_RETRIEVAL`、`VECTOR_CANDIDATE_K`

仓库必须维护一个版本化的示例配置文件：`backend/.env.example`。

## 9. 非功能需求

- 项目应可在 Windows、macOS 和 Linux 上本地启动。
- 项目运行不依赖本地 GPU。
- 代码应支持在 mock LLM 和 mock 向量库条件下测试。
- 文档必须反映当前生效架构，不能继续描述已经废弃的实现。
- CI 必须分别运行后端和前端测试。

## 10. 仓库文档要求

仓库至少应维护以下文档：

- `README.md`：首次打开仓库时的总览和启动说明。
- `CONTRIBUTING.md`：协作方式、测试要求和提交流程。
- `backend/.env.example`：配置项模板。
- `docs/requirements.md`：当前产品与工程需求基线。
- `backend/openapi.json`：后端接口契约变更后的导出结果。

## 11. 当前不纳入范围

当前基线不要求以下内容：

- 多租户云部署
- 细粒度 RBAC
- 分布式生产级存储
- 原生桌面打包
- 在未配置外部模型提供方时仍完全独立运行
